# Мудрости гориллы

Локальное веб-приложение: пользователь задаёт вопрос, выбирает один из 11 уровней мудрости и получает ответ от OpenAI-совместимой модели, запущенной на том же компьютере или сервере. Браузер обращается только к FastAPI backend; URL модели, ключ и системный промт не попадают на клиент.

## Требования

- Python 3.10+
- локальный сервер модели с OpenAI-совместимым Chat Completions API

Например, подойдут локальные серверы, которые предоставляют `POST /v1/chat/completions` и принимают поля `model`, `messages`, `temperature`.

## Установка

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
Copy-Item .env.example .env
```

Откройте `.env` и настройте модель:

```dotenv
LLM_BASE_URL=http://127.0.0.1:8001/v1
LLM_MODEL=local-model
LLM_API_KEY=
LLM_TIMEOUT_SECONDS=120
MAX_QUESTION_LENGTH=4000
```

`LLM_BASE_URL` должен оканчиваться на базовый путь API (`/v1`), а не на `/chat/completions`. Пустой `LLM_API_KEY` допустим для локальных серверов без авторизации.

Системный промт находится в `prompts/silverback-system.txt`. Backend читает его при запуске и передаёт локальной модели первым сообщением с ролью `system`; пользовательский вопрос всегда идёт следующим сообщением с ролью `user`.

## Автоматическая установка на Ubuntu с Qwen2.5 0.5B Q4

Скрипт `setup-ubuntu.sh` устанавливает Python-окружение, Ollama, модель `qwen2.5:0.5b-instruct-q4_K_M`, настраивает `.env` и создаёт два автоматически запускаемых systemd-сервиса: Ollama и сайт.

Запускайте его из корня проекта. Поддерживается как обычный пользователь с доступом к `sudo`, так и непосредственный запуск от `root`:

```bash
chmod +x setup-ubuntu.sh
./setup-ubuntu.sh
```

При запуске от `root` исходники безопасно копируются в `/opt/monkey-wisdom`, а сам сайт работает не от root, а от отдельного системного пользователя `monkey-wisdom`.

По умолчанию сайт слушает только `127.0.0.1:8000`. Для доступа из локальной сети:

```bash
SITE_HOST=0.0.0.0 ./setup-ubuntu.sh
```

Для изменения порта можно передать `SITE_PORT`, например `SITE_PORT=8080 ./setup-ubuntu.sh`. Скрипт рассчитан на Ubuntu с systemd и требует интернет во время установки пакетов и загрузки модели (около 398 МБ).

### Публикация по IP через Nginx

После успешного выполнения `setup-ubuntu.sh` запустите:

```bash
chmod +x setup-nginx-ip.sh
./setup-nginx-ip.sh
```

Скрипт настраивает Nginx на всех сетевых интерфейсах (`0.0.0.0:80` и `[::]:80`), оставляет FastAPI и Ollama закрытыми на localhost, открывает текущий SSH-порт и HTTP-порт в UFW и выводит адрес вида `http://ПУБЛИЧНЫЙ_IP`. Его также можно запускать через `sudo` или непосредственно от `root`.

Если SSH работает на нестандартном порту и скрипт запущен не из SSH-сессии, передайте порт явно:

```bash
SSH_PORT=2222 ./setup-nginx-ip.sh
```

## Запуск

Убедитесь, что локальная модель запущена, затем выполните:

```powershell
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

Откройте [http://127.0.0.1:8000](http://127.0.0.1:8000). Swagger-схема доступна на `/docs`.

Для доступа из локальной сети:

```powershell
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000
```

При публикации на `0.0.0.0` сайт становится доступен другим устройствам сети. Разрешайте входящие подключения в firewall только для доверенной локальной сети; приложение не предназначено для прямого публичного доступа в интернет.

## API

Запрос:

```powershell
$body = @{ question = 'Как принять трудное решение?'; wisdomLevel = 0.8 } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri http://127.0.0.1:8000/api/wisdom -ContentType 'application/json' -Body $body
```

Ответ:

```json
{"wisdom":"..."}
```

`wisdomLevel` принимает значения от `0` до `1` с шагом `0.1`. Backend вычисляет температуру сам: `temperature = round(1 - wisdomLevel, 1)`. Клиентская температура никогда не принимается.

Проверка состояния приложения и модели:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/health
```

- HTTP 200: приложение и endpoint `/models` локальной модели доступны.
- HTTP 503: приложение работает, но локальная модель недоступна.

## Тесты

```powershell
pip install -r requirements-dev.txt
pytest -q
```

Тесты не требуют запущенной модели: её API имитируется локально. Они проверяют все 11 соответствий уровня и температуры, валидацию, порядок `system`/`user`, сбои модели, пустой ответ и состояния интерфейса.

## Ограничения и защита

- Вопрос ограничен `MAX_QUESTION_LENGTH` символами, а размер HTTP-запроса — связанным безопасным пределом.
- На один IP разрешено 12 запросов в минуту; лимит хранится в памяти процесса.
- Ответ модели вставляется в страницу через `textContent`, а не как HTML.
- Системный промт, внутренний URL и ключ не возвращаются в API и не логируются приложением.
- Health-check проверяет стандартный OpenAI-совместимый endpoint `/models`; сервер модели, который его не реализует, будет показан как недоступный, даже если Chat Completions работает.
